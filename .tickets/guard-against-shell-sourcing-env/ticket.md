## What we need

A warehouse-level convention (and ideally a guardrail) that stops scaffolded-project agents from loading `.env` by shell-sourcing it. Two concrete asks:

1. **Document the safe pattern in the scaffolded templates.** Add to the template `CLAUDE.md` and/or `docs/reference/conventions.md` a short, explicit rule such as:

   > **Never `source`/`. `/`set -a` a `.env` file in the shell.** Its values may contain `&`, spaces, or other shell metacharacters; `source` executes them as shell code, which can run fragments as background jobs and **echo the values (including secrets) into the session log**. Load `.env` via `python-dotenv` (the app already does this in its entry points). For an ad-hoc shell need, run a `python -c` that calls `load_dotenv()` and uses the value — never `source`.

2. **Consider a shared agent guardrail.** A note in a session-start/maintenance skill (or a settings hook that warns on `source .env` / `. .env` / `set -a; . .env` in Bash commands) so the footgun is caught before it fires, not just documented.

3. **(Optional)** Have the scaffolded `.env.example` quote its placeholder values to model safe handling — though "load via python-dotenv only" is the cleaner guidance and makes quoting moot.

## Why we need it

While building the parameter-range validator in WaterQualityFeed, the agent needed an env var for a one-off live-pull acceptance script and loaded `.env` with `set -a; . "$MAIN/.env"; set +a`.

WaterQualityFeed's `.env` holds HTTP-trigger SAS URLs for Power Automate proxy flows (`WQ_PROXY_FLOW_URL`, `WQ_WRITER_FLOW_URL`) — and **the URL itself is the bearer credential**. Those URLs are unquoted and contain `&` (`...?api-version=1&sp=...&sv=1.0`). When sourced, the shell:

- read only the fragment before the first `&` as the assignment,
- ran `sp=...` and `sv=1.0` as **background jobs**, and
- job-control printed each full command line — including the SAS `sig` tokens — to stdout as `[1] Done ...`.

Result: both proxy URLs leaked into the session log and had to be rotated.

This was **not** a project-specific defect — the safe path already exists and is used by every real entry point (`run_pipeline.py`, `scripts/clone_canonical_to_sp.py`, `tests/live/conftest.py` all call `python-dotenv`'s `load_dotenv()`, which parses `&` correctly and never echoes). Nothing steered the ad-hoc command away from `source`. Because the warehouse scaffolds the `.env` + conventions pattern into every target project, the same footgun is latent in any scaffolded repo whose `.env` holds unquoted URL/secret values — so the fix belongs here, not in one target project.
