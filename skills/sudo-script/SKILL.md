# Sudo Script

When a task needs the user to run privileged or interactive commands the harness can't execute (sudo, package installs, system service control, anything that prompts), don't ask the user to paste a series of commands. Write the commands to one script, give them one invocation, then verify and clean up.

This skill is interactive in the sense that it requires the user to run a command outside the conversation. If auto mode is active and no human is available to run it, **stop and surface** what the script would do, then exit.

## Process

### 1. Compose the script

Pick a unique-ish path so concurrent invocations don't collide:

```
/tmp/_run_<8-hex-chars>.sh
```

Write the script with:

- A `#!/usr/bin/env bash` shebang.
- `set -euo pipefail` at the top.
- A short header comment naming the purpose, generated by, and date.
- The commands, one per line, with brief comments for each block.
- An `echo "OK"` (or similar success indicator) at the end so the user can confirm completion.

Make it executable: `chmod +x /tmp/_run_<hex>.sh`.

### 2. Tell the user

Show:

- **What the script does** — one or two sentences.
- **The commands inside** — paste the full script body so the user can review before running.
- **One invocation** — the single command to copy-paste:

```
$ bash /tmp/_run_<hex>.sh
```

If sudo is needed, tell the user: "This needs sudo — run with `sudo bash /tmp/_run_<hex>.sh`."

### 3. Wait for the user to run it

Wait for the user to confirm completion (and paste any output if relevant).

In auto mode without a user, **do not write the script** — surface what it would have contained and stop. The user runs it on return.

### 4. Verify

If the script's effects are observable (a new file exists, a service is running, a package is installed), verify:

```bash
which <installed-binary>
systemctl is-active <service>
ls <expected-file>
```

Confirm to the user that the change took effect.

If verification fails, surface what was expected vs observed and ask for the script's full output.

### 5. Clean up

Delete the script:

```bash
rm /tmp/_run_<hex>.sh
```

Use plain `rm` (not `rm -rf`, which is denied by warehouse settings).

### 6. Report

- **What ran:** one-line summary.
- **Verification:** what was checked.
- **Cleaned up:** confirmation that the script is gone.

## When to use this vs running directly

Use this skill when:

- The command needs sudo and the harness denies it.
- Multiple commands need to run in sequence as the user (and copy-pasting them one at a time is friction-heavy).
- An interactive command (e.g. an installer that prompts) needs the user's terminal.

Don't use this skill when:

- A single command can run in the agent's harness.
- The change is reversible and low-risk and the harness allows it.

## Installation

This skill lives in `~/.claude/skills/sudo-script/` (per-user). It's not symlinked from a project's `.claude/skills/` because it applies globally — every project benefits from having it available.

Install once:

```bash
ln -s /home/<user>/AgenticEngineering/skills/sudo-script ~/.claude/skills/sudo-script
```

Or copy if portability is needed.

## What this skill does NOT do

- Does not run sudo or privileged commands inside the harness — that's why this skill exists.
- Does not silently leave scripts in `/tmp/` — always cleans up.
- Does not bypass user confirmation — the user explicitly runs the script.
- Does not write to system files or rotate logs without the user's involvement.
