# Triage

Move tickets through a small state machine of canonical roles. Produce durable agent briefs for tickets that move to `ready-for-agent`.

Every comment or update posted by this skill must start with the disclaimer:

```
> *This was generated by AI during triage.*
```

## Refuse auto mode

If auto mode is active, respond:

> Triage requires your direction on each ticket — switch to interactive mode and re-invoke `/triage`.

Then exit.

## State machine

Five **state roles**:

- `needs-triage` — maintainer needs to evaluate.
- `needs-info` — waiting on reporter for more information.
- `ready-for-agent` — fully specified, AFK-ready (an agent can pick up cold).
- `ready-for-human` — needs human implementation (judgment, external access, design call).
- `wontfix` — will not be actioned.

Two **category roles**:

- `bug` — something is broken.
- `enhancement` — new feature or improvement.

A triaged ticket carries exactly one state and one category.

Canonical transitions: unlabeled → `needs-triage` → (`needs-info` ↔ `needs-triage`) → (`ready-for-agent` | `ready-for-human` | `wontfix`).

## Invocation

The user invokes `/triage` and describes the intent in natural language. Examples:

- "Show me anything that needs my attention"
- "Let's look at #42"
- "Move #42 to ready-for-agent"
- "What's ready for agents to pick up?"

Interpret and act. For free-form requests, present three buckets oldest-first:

1. **Unlabeled** — never triaged.
2. **`needs-triage`** — evaluation in progress.
3. **`needs-info` with reporter activity since the last triage notes.**

Show counts and a one-line summary per ticket. Let the user pick.

## Triaging a specific ticket

### 1. Gather context

Read the full ticket (body, comments, status, dates). Parse any prior triage notes so you don't re-ask resolved questions. Lightly explore the codebase using `glossary.md` and respect ADRs in the touched area. Read `.out-of-scope/*.md` if present and surface any prior rejection that resembles this ticket.

### 2. Recommend

Tell the user your category and state recommendation, with reasoning, plus a brief codebase summary relevant to the ticket. Wait for direction.

### 3. Reproduce (bugs only)

Before any grilling, attempt reproduction: read the reporter's steps, trace the relevant code, run tests or commands. Report what happened — successful repro with code path, failed repro, or insufficient detail (a strong `needs-info` signal). A confirmed repro makes a much stronger agent brief.

### 4. Grill (if needed)

If the ticket needs fleshing out, run a `/grill` session inline.

### 5. Apply the outcome

- `ready-for-agent` — post an agent brief comment using the template below.
- `ready-for-human` — same template, but note explicitly why a human is needed.
- `needs-info` — post triage notes (template below).
- `wontfix` (bug) — polite explanation, then close.
- `wontfix` (enhancement) — write a `.out-of-scope/<concept>.md` file (lazy-create the directory) capturing the decision and reasoning, link it from a comment, then close.
- `needs-triage` — apply the role; optional comment if there's partial progress.

### 6. Quick state override

If the user says "move #42 to ready-for-agent", trust them. Apply the role, confirm what you'll do (role change, comment, close), then act. Skip grilling. If moving to `ready-for-agent` without grilling, ask whether to write an agent brief.

## Agent brief template

```md
> *This was generated by AI during triage.*

## Agent Brief

**Category:** bug | enhancement
**Summary:** one-line description of what needs to happen.

**Current behaviour:**
What happens now.

**Desired behaviour:**
What should happen after the work is complete. Be specific about edge cases and error conditions.

**Key interfaces:**
- `TypeName` — what needs to change and why.
- `functionName()` return type — current vs. desired.
- Config shape — any new options needed.

**Acceptance criteria:**
- [ ] Specific, testable criterion 1
- [ ] ...

**Out of scope:**
- Things NOT addressed in this ticket.
```

**Durability rules** (the brief outlives the current code state):

- Describe interfaces, types, behavioural contracts. **Not** file paths or line numbers.
- Behavioural ("the type should accept X"), not procedural ("open file Y line Z and edit").
- Each acceptance criterion independently verifiable.

## Triage notes template (`needs-info`)

```md
> *This was generated by AI during triage.*

## Triage Notes

**What we've established so far:**

- point 1
- point 2

**What we still need from you (@reporter):**

- specific question 1
- specific question 2
```

Capture everything resolved during grilling under "established so far" so the work isn't lost. Questions must be specific and actionable.

## Resuming a previous session

If prior triage notes exist on the ticket, read them, check whether the reporter has answered any outstanding questions, and present an updated picture. Don't re-ask resolved questions.

## What this skill does NOT do

- Does not write code.
- Does not modify the original ticket body — appends comments and updates status only.
- Does not skip the disclaimer line on any AI-generated comment.
