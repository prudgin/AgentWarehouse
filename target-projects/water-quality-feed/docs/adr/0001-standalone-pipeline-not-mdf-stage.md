# Standalone pipeline, not an MDF stage

WaterQualityFeed could have been a new stage inside MercatusDataFeed — MDF already ingests external SharePoint data inline (ledger step `10e_link_sample_weights_to_cycles.py` pulls sample weights via rclone) so there is precedent. We make it **standalone** anyway. Three reasons. (1) MDF's inline pattern is for data consumed by exactly one ledger step; WQ is a broader dataset with multiple plausible consumers (MDF analysis, Power BI, ad-hoc analyses), and a separate publish root (`/mnt/data/water_quality/`) is the right boundary. (2) Different upstream, different failure modes — Mercatus OData and the Power Automate HTTP proxy fail independently (auth lifetime, throttling, Flow runtime), and co-locating means one outage can take both pipelines down. (3) WQ readings are `(UnitId, timestamp)`-keyed and don't need cycles to exist; cycle attachment is a *downstream join*, not an *ingest concern*. The cost: own systemd timer, own healthcheck, own `.env`. Worth it for the isolation. The "compatible with MDF cycles" requirement is preserved as a schema contract (Sydney TZ, parquet, MDF-style column conventions) without requiring co-located code.
