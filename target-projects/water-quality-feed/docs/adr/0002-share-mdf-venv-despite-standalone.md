# Share MDF's `.venv` despite being a standalone project

This appears to contradict [ADR-0001](0001-standalone-pipeline-not-mdf-stage.md) — if the point of standalone is failure isolation, why couple Python environments? Because the two boundaries are different. ADR-0001 is about **publish-contract and run-isolation**: WaterQualityFeed has its own publish root, its own systemd timer, its own healthcheck, its own `.env`, its own git repo. A failure in WQ does not prevent MDF from running and vice versa. The shared venv only couples **package installation state**, which is the cheap part — when MDF's venv would be broken, the operator is already aware (because MDF is broken too) and fixing it fixes both. The argument *for* sharing: sibling consistency. The venv at `~/PycharmProjects/MercatusDataFeed/.venv/` already hosts MDF, GrowthModels, PowerBI, and FishGrowthFittingSGRpackage; making WaterQualityFeed the 5th tenant means one Python install to keep current rather than two on the same machine, and zero risk of pandas/pyarrow version drift if a future cross-project stage (e.g. the canonical-mapping stage) ever does want to read MDF's parquets with the same library versions MDF wrote them with. Knock-ons: WQ has a `pyproject.toml` with bare-name deps (inheriting MDF's no-version-pins rule), is `pip install -e`'d into MDF's venv, and the systemd unit's `ExecStart` points at MDF's venv interpreter.
