# Scripts colocate with their owning skill

The warehouse ships executable scripts (e.g. `check-docs.sh` for the orphan and broken-link sweep). They could live in a top-level `scripts/<name>/` directory parallel to `skills/`, `templates/`, and `target-projects/` — or inside the skill that owns them at `skills/<skill>/scripts/<name>.sh`. We chose colocation. Reasons: (a) it avoids introducing a third warehouse category alongside library (declarative) and skills (procedural) per [ADR-0001](0001-library-and-skills-coexist.md) — scripts are an implementation detail of the skill, not a peer; (b) project symlinks of the skill directory bring the script along for free, no extra symlink machinery; (c) skills become self-contained — a skill's `SKILL.md`, its scripts, and any reference material live together, and a maintainer reading one skill sees the whole thing in one place. Trade-off: a script that legitimately serves two skills must pick a primary owner, with the secondary skill's `SKILL.md` referencing it by relative path. Cost is low — most scripts are tightly bound to one skill, and a shared script just means an explicit cross-link.
