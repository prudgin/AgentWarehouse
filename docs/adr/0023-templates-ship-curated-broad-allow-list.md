# Templates ship a curated broad allow-list with a tight deny floor

Each template's `.claude/settings.json` had only two `rm -r*` denies and no allows. New projects scaffolded from that baseline had too-much-friction at startup — every routine operation (file copies, git status, python invocations, jq parses) hit a permission prompt. We move templates to a **curated broad allow-list above a tight deny floor**: broad enough that routine work runs without prompting, narrow enough that destructive operations always confirm. The shape: a common base shared by all four templates (read-only bash, safe file ops, safe git read + add/commit/init/switch/checkout-b, Python tooling, common Unix utilities), plus per-template additions where the template's nature requires them — `tool-integration` adds `Bash(_tools/*)` execution patterns; the other three need no additions beyond the base. The deny floor is uniform across templates: `rm -rf*`, `rm -r*`, `rm /*`, `sudo*`, force-push variants, `git reset --hard*`, `git clean -f*`, hard checkout/restore variants, `git rebase*`, `git filter-branch*`, `chmod 777*`, and `curl|wget … | sh|bash` patterns. Trade-offs: (a) broad allows mean some operations agents take by mistake won't prompt — but the deny floor catches the destructive-by-mistake cases, and recoverable mistakes (a stray file write, an unwanted commit) are reversible via the agent's own subsequent operations or git. (b) Per-project override stays cheap — edit `.claude/settings.json` or use `.claude/settings.local.json`. (c) Different projects have different threat models; a security-sensitive project should tighten the inherited defaults rather than build up from minimal — that's the right starting position. Rejected alternatives: minimal-denies-only (proven friction-y in real use), single shared allow-list across templates with no per-template tuning (forces tool-integration to either add execution allows in its CLAUDE.md as a special case or live without them).
