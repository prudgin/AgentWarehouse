# Tool-integration surfaces have a README; `/finish` enumerates per-artifact dirs

Tool-integration projects accumulate per-artifact directories under each platform surface (e.g., `tasks/Write_quarterly_report/` under `tasks/`, `flows/Daily_export/` under `flows/`). Each artifact dir contains a `*-meta.json` recording the platform GUID, display name, and export provenance. We pin two conventions: **(1)** every surface directory carries a `README.md` that documents the surface's conventions (display-name normalisation, meta-field semantics, push-vs-pull policy) but does **not** list specific artifacts — surface READMEs stay stable while artifacts churn. **(2)** `/finish` gains an enumeration step for tool-integration projects: for each surface dir referenced in CLAUDE.md's documentation map, walk every `<Name>/` subdir and verify the surface-specific meta file exists; surface a missing or malformed meta as an integrity error. Per-artifact READMEs are deliberately **not** required — the meta.json is the source of truth for "is this dir well-formed." Trade-offs: (a) a surface README is one more file to maintain per surface (small and stable, low cost); (b) the alternative of "no README, convention-only" leaves no clear home for surface-specific quirks, which would otherwise leak into CLAUDE.md or skill bodies; (c) the alternative of "agent-discipline only, no `/finish` enforcement" was rejected because the self-test surfaced exactly this gap and per-project drift is high-likelihood. Only applies to projects scaffolded from the tool-integration template — other templates have no surface concept.
