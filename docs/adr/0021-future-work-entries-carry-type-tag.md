# Future-work entries carry an explicit `**Type:**` tag

`docs/planning/future-work.md` entries already carry structured fields: `**What:**`, `**Why:**`, `**Open questions:**`, `**Links:**`. We add `**Type:** watching | open-question | proposal | refinement-candidate` as an additional required field. `/finish` step 6 (future-work graduation sweep) then collapses from judgment-per-entry-against-three-heuristics to a deterministic grep: any entry tagged `proposal` is a graduation candidate; the other three types stay. The self-test confirmed the heuristics work for small entry counts (6 out of 6 classified correctly), but heuristic application is per-agent judgment that grows linearly with backlog size, while a grep is constant. Tagging also serves as a discipline at write-time: if you can't classify an entry under one of the four types, the entry is fuzzy and shouldn't be in `future-work.md` yet — write it elsewhere or sharpen first. Cost: one-time mechanical migration across the warehouse + four templates' `future-work.md`, plus one extra line per new entry. Rejected alternatives: keep heuristic-based (works now, scales poorly), and hybrid tag-with-heuristic-fallback (maintains two code paths in step 6 and dilutes the discipline). The four type values are exhaustive enough — *proposal* is the only graduating type; *watching* / *open-question* / *refinement-candidate* are the three patterns that legitimately stay in `future-work.md` even when active.
