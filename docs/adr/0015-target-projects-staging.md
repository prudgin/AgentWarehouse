# Target-projects staging mirrors eventual project layout, with `_warehouse/` for scratch

The warehouse needs a place to accumulate decisions about a target project before they get written into the target's actual repo. Staging lives in `target-projects/<name>/` and **mirrors the eventual project layout**: `glossary.md`, `CLAUDE.md`, `docs/adr/`, `docs/domain/`, `docs/planning/future-work.md`. `migrate-project` and `create-project` transfer everything outside `_warehouse/` to the target repo. Warehouse-side scratch — intake notes, migration plan, status, post-handoff feedback — lives in `_warehouse/<file>.md`, cordoned off by the underscore prefix and skipped at transfer time. Mirroring keeps migration mechanical and verifiable (a `diff` of staged vs. transferred is trustworthy); the `_warehouse/` carve-out keeps warehouse meta-state from polluting the project draft. The flat-workspace alternative (skill maps decisions to project layout at transfer time) was rejected because the mapping logic becomes an opaque step that defeats easy verification, and because looser staging structure invites drift across target projects. Per-target dirs are **permanent**: completed entries persist as institutional memory of every project the warehouse has set up.
